# 模型路由与二次确认协议

## 1. 目标

本项目在让 AI 协助开发时，不希望每次都重复解释：

- 现在做哪个阶段
- 用什么模型
- 为什么用这个模型
- 是否要二次确认

因此约定一个固定流程：

```text
先识别任务
→ 推荐模型
→ 用户二次确认
→ 再执行
```

## 2. 模型使用原则

### GPT-5.4 Mini

适合：

- 文档总结
- 学习讲解
- 面试问答整理
- 简历文案
- 方案归纳

特点：

- 便于高频使用。
- 适合低风险、低复杂度、信息整理类任务。

### Composer 2.5

适合：

- 日常编码
- 页面开发
- 组件封装
- 路由和布局
- 普通业务逻辑

特点：

- 适合项目内边读边改。
- 适合作为编码阶段默认主力模型。

### Auto

适合：

- 不确定难度的 bug
- 不确定复杂度的任务
- 不知道该选哪个模型时先分诊

特点：

- 作为自动路由兜底。
- 适合先试一轮，再决定是否切强模型。

### GPT-5.5 / Codex 5.3

适合：

- 跨文件复杂改动
- 报表中心复杂状态
- 性能优化
- AI Agent 业务逻辑
- 难度较高的 Vue3 + TS 组件设计

特点：

- 更适合复杂任务和较大上下文推理。

### Opus 4.8

适合：

- 极难的 3D / Cesium / WebGL / 架构问题
- GPT-5.5 / Codex 仍然不够稳时的专家会诊

特点：

- 少量、集中使用。
- 不建议日常高频用。

## 4. 二次确认协议

当用户说：

```text
按 Dining Ops Platform 项目规则，开始第 X 阶段
```

AI 应先输出：

1. 当前阶段名称
2. 推荐模型
3. 推荐原因
4. 备选模型
5. 询问用户是否使用推荐模型，还是改用其他模型

### 推荐回复模板

```text
当前准备开始第 X 阶段：[阶段名称]。

推荐模型：Composer 2.5
原因：这个阶段主要是 [页面/组件/布局/常规编码]，Composer 2.5 更适合边读边改和项目内多文件实现。

备选模型：
- Auto：如果你希望先让模型自动分诊
- GPT-5.5：如果你希望我先做更强的方案推理

你要我按推荐模型继续，还是换成别的模型？
```

## 5. 何时直接升级到强模型

如果出现以下情况，不要先耗一轮轻模型：

- 报表状态设计已经复杂。
- 权限和组织穿透比较绕。
- Cesium / WebGL / 3D 问题。
- 性能问题已经定位到复杂根因。
- AI Agent 需要跨模块设计。
- 连续两轮 Auto 或 Composer 仍无法稳定解决。

此时建议直接切：

```text
GPT-5.5 / Codex 5.3 / Opus 4.8
```

## 6. 不建议的使用方式

- 不要所有任务都默认 Auto。
- 不要复杂任务只靠 GPT-5.4 Mini。
- 不要一上来就用 Opus。
- 不要让强模型只负责文档总结，浪费额度。

## 7. 最终策略

```text
文档总结 → GPT-5.4 Mini
日常开发 → Composer 2.5
不确定任务 → Auto
复杂架构 / 3D / 报表 / AI Agent → GPT-5.5 / Codex 5.3
特别难的 3D / Cesium / WebGL → Opus 4.8
```

## 8. 使用建议

以后你可以直接说：

```text
按 Dining Ops Platform 项目规则，开始第 5 阶段。
```

然后让 AI 按本协议：

- 先推荐模型
- 再二次确认
- 再执行

这样就能把模型搭配使用做到相对最优。
